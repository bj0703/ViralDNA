## Context

当前emo_transfer项目5Agent流水线已完整运行，但存在致命Bug：FFmpeg渲染层从timeline segment读取`asset`字段，但上游所有Agent输出的字段是`asset_id`，导致全部片段静默回退到索引0（恰好是参考样例视频），最终产出的视频是样例的重复剪辑，完全没有使用素材。同时后续要对接前端可视化剪辑轨道界面，需要在数据结构层预留足够信息支持多轨道可视化渲染。

现有约束：
- 不能破坏向后兼容性
- 不能引入外部大依赖
- 保持现有流水线拓扑不变
- 渐进式增量升级

## Goals / Non-Goals

**Goals:**
1. 修复当前致命渲染Bug，确保流水线真正从素材视频中取片段拼接出片
2. 扩展edit_timeline数据模型，完整预留多轨道可视化所需的全部字段
3. 新增Timeline Editor中间层服务，提供原子操作API供后续前端调用
4. 增加前置校验层，彻底消除静默失败回退到索引0的隐患
5. 100%向后兼容，现有调用方无需修改任何代码即可正常运行

**Non-Goals:**
1. 本次不实现前端UI界面，只预留后端能力
2. 不引入复杂的第三方剪辑库依赖
3. 不重构现有5Agent流水线的拓扑和职责划分
4. 本次不实现音频波形可视化和逐帧预览播放组件

## Decisions

| 决策 | 方案 | 理由 | 替代方案 |
|------|------|------|----------|
| 字段兼容策略 | 同时读取`asset`和`asset_id`两个字段，任一存在即可 | 最小侵入性修复，上游Agent不需要改任何代码就能正常工作 | 强制上游统一改字段为asset，改动量大风险高 |
| 参考视频过滤策略 | FinalVideoRenderer传给FFmpeg的source_paths只包含`is_reference=False`的素材 | 从根源上避免样例视频被意外引用，即使索引0错误也碰不到样例 | 保留全部视频在池子里，靠校验排除，不如彻底过滤安全 |
| 数据模型扩展策略 | 新增顶层`tracks[]`多轨道数组，保留原有扁平`timeline[]`作为兼容别名 | 完全向后兼容，旧渲染器继续可用，前端新逻辑直接用新tracks结构 | 直接修改旧timeline结构为嵌套多轨道，破坏所有现有依赖 |
| 素材引用定位策略 | 新增`asset_full_path`字段直接携带绝对路径，不依赖文件名哈希映射 | 彻底避免文件名重名冲突问题，前端拿到字段直接可以打开素材预览 | 维护全局asset_id→path映射表，增加复杂度和查找开销 |
| 时间→像素换算策略 | 后端预先计算`timeline_start_px`和`timeline_end_px`并存入segment | 前后端渲染逻辑完全一致，前端零计算直接渲染轨道条 | 前端自行根据画布宽度换算，容易出现像素对齐偏差 |
| 校验层策略 | 在FFmpeg渲染入口前新增独立validate_timeline()函数 | 集中式校验所有引用合法性，一处修改全局受益 | 分散在各Agent各自校验，逻辑重复易遗漏 |

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 扩展后timeline数据体积膨胀 | 只在tracks结构中保留扩展字段，旧timeline数组保持精简，JSON序列化总增量控制在30%以内 |
| 向后兼容出问题 | 完整保留旧字段路径，新增单元测试覆盖新旧数据结构所有分支 |
| 预留字段过度设计后续用不上 | 所有新增字段都是可选的，下游逻辑读取时设合理默认值，不存在强制依赖 |
| timeline编辑服务并发冲突 | 共享内存天然是单job单用户模式，不需要复杂的锁机制，简单校验变更前后hash即可 |
