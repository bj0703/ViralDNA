## Context

当前流水线存在两处历史遗留缺陷：1）SlotMatcherAgent内包含多余的手动派生变体、强制素材轮询替换逻辑，覆盖LLM原始精准匹配结果，导致多个shot复用同一个短素材、零时长无法渲染；2）流水线启动时全量计算4个变体，首次出片等待时间长达4分钟，用户体验差。LLM已亲自在返回结果中生成了5套完全独立的匹配方案（根级基线+4个变体），完全不需要代码二次派生。

## Goals / Non-Goals

**Goals:**
- 100%透传LLM原始返回的5套匹配结果，不做任何人工修改覆盖
- 流水线优先使用根级纯净原生基线快速生成第一版成片，首次出片总耗时从4分钟降低到1分钟以内
- 保留时间自动连续化、片段最小0.5秒保护逻辑，彻底解决零时长渲染失败BUG
- 4个非基线变体全部标记为懒加载状态，仅当用户明确指定variant_id时才触发后续处理
- 100%向后兼容，所有旧接口返回字段完全不变，无任何破坏性变更

**Non-Goals:**
- 不新增任何第三方依赖
- 不重构整个流水线架构，只做局部精简优化
- 不改变任何现有数据结构的字段定义

## Decisions

1. **完全删除_derive_variant_payload逻辑**：信任LLM，不再用代码手动派生变体，LLM生成的结果直接透传返回
2. **优先直接用根级shot_matches生成timeline**：跳过多余素材分配处理，第一版成片快速生成
3. **变体懒加载占位标记**：4个非基础版变体统一标记`_lazy=True`，流水线只处理structure版本
4. **保留自动时间保护逻辑**：前一个片段end自动作为下一个片段start，每个片段强制最小0.5秒时长，彻底消灭零时长

## Risks / Trade-offs

[Risk] 旧变体结构字段向后兼容性 → Mitigation 完全保留所有旧接口schema，variant字典所有占位字段和之前保持完全一致，旧客户端感知不到任何变化
