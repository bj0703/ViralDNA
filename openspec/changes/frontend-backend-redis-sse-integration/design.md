## Context

当前项目前端已有完整的三栏UI组件布局（左侧视频资源区、中间播放器+时间线、右侧Agent对话区），后端已具备 Agent 编排流水线、Redis 共享内存存储和 SSE 事件流基础能力。但前后端之间缺少一层完整的对接层：前端状态硬编码在本地 mock 中，没有与后端 Redis 结构化数据绑定；SSE 连接没有统一管理；Agent 输出内容没有自动映射到三个 Tab 区域展示。本变更将基于《前后端数据操作流程.md》补齐这层对接。

## Goals / Non-Goals

**Goals:**
- 实现 useSSEEventSource Hook，统一管理所有 SSE 连接生命周期
- 前后端核心数据类型 100% 对齐，零字段转换即可直接传递
- 视频资源区、时间线、Agent 对话区三大核心链路实时数据打通
- 多用户协作场景下画面秒级同步
- 向后完全兼容原有 REST API，Redis 不可用时自动降级到内存模式

**Non-Goals:**
- 不重写任何已有的 Agent 业务逻辑
- 不修改后端现有的 Redis 数据结构设计
- 不重构已完成的 UI 组件视觉样式
- 不引入除了浏览器原生 EventSource 之外的第三方 SSE 库

## Decisions

1. **SSE Hook 独立封装为自定义 Hook**
   - 选择：在 `frontend/src/hooks/useSSEEventSource.ts` 中完全自包含实现
   - 不引入 socket.io 等第三方库，使用浏览器原生 EventSource
   -  rationale：减少依赖体积，避免额外协议开销，断点续传通过 lastEventId 原生支持
   - 重连策略：指数退避 1s → 2s → 4s → 8s，超过 3 次失败自动降级为 1.5s 轮询兜底

2. **类型定义集中在单个 types/index.ts 文件**
   - 选择：所有前后端共享的类型定义集中在 `frontend/src/types/index.ts`
   - 完全复用后端字段命名，不做重命名映射
   - rationale：避免前后端字段名不一致导致的隐性 bug，前端直接映射后端返回的 JSON，零转换成本

3. **Agent对话区初始化策略：打开即干净空白 + 草稿箱手动加载历史**
   - 选择：页面刚打开时对话区完全清空，不自动加载任何历史对话/历史事件
   - SSE 连接建立成功后，只实时接收**新产生的增量事件**
   - 新增草稿箱入口，用户主动点击后才展示该 Job 对应的历史对话事件
   - rationale：保证刚打开页面体验清爽干净，新用户不受之前历史内容干扰，聚焦当前操作

4. **编辑操作乐观 UI 更新 + 后端最终确认**
   - 选择：用户拖拽/修改片段时前端先立即更新本地画面，同时派发 PATCH 请求到后端，后端返回后确认状态一致性
   - rationale：操作无延迟感知，体验流畅，即使网络抖动也不影响本地交互

5. **向后兼容保障：所有原有 REST API 完全保留**
   - 选择：SSE 为增强能力而非强制依赖，后端自动检测 Redis 可用性，不可用时切到本地内存模式，所有接口行为不变
   - rationale：系统零单点故障，服务 100% 可用不中断

## Risks / Trade-offs

- [Risk] SSE 在某些企业网络环境下被防火墙拦截 → Mitigation：Hook 内置自动降级到轮询兜底机制，对上层组件完全透明
- [Risk] SSE 断线重连期间产生的事件丢失 → Mitigation：使用 Last-Event-ID 头做断点续传，重连后自动拉取断点后的所有事件
- [Risk] 前后端类型定义不一致导致字段不匹配 → Mitigation：新增统一 ZOD Schema 校验层做运行时数据校验，不匹配时打印明确的 console.error 日志
- [Risk] 多用户同时编辑同一时间线片段产生冲突 → Mitigation：后端已有分布式 RedLock 保证原子操作，前端通过 SSE 广播最新状态覆盖本地冲突修改
