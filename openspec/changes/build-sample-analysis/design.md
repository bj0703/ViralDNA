## Context

PRD 建议按照“样例理解 -> 结构抽取 -> 素材适配 -> 结果生成”四步来搭建整个系统。当前仓库里只有一个早期原型 [video_upload.py](/d:/ai%20coding/emo_transfer/video_upload.py)，它能够上传视频并请求多模态模型做分析，但目前还没有一个明确的产品契约来规定“样例分析”到底必须产出哪些内容。

这次 change 的目标就是先定义第一份正式契约，并且刻意控制 scope。我们希望先做出一个稳定、可展示、可复用的样例分析阶段，让它既能单独演示，也能为后续 change 提供输入。

## Goals / Non-Goals

**Goals:**
- 定义系统接收一个或多个样例视频时的最小产品行为。
- 把分析输出标准化为可复用字段，而不是只返回自由文本总结。
- 第一阶段至少支持两类结构维度的抽取：脚本结构和节奏结构。
- 让分析结果对评审和开发都足够直观，能够看到明确的中间结果。

**Non-Goals:**
- 不在这一阶段生成新脚本、分镜或成片视频。
- 不在这一阶段处理目标素材的缺口识别。
- 不在这一阶段生成字幕包装、贴纸、封面等包装建议。
- 不在这一阶段把实现绑定到某一个固定模型厂商，只保留与现有原型兼容的开放性。

## Decisions

### 1. 将样例分析作为独立 capability
第一阶段只定义一个能力 `sample-analysis`，不把结构迁移或编辑能力一起塞进来。

原因：
- 这和 PRD 推荐的第一步一致。
- 这样可以显著降低第一轮实现风险，尽快拿到可演示成果。
- 后续阶段可以依赖一个稳定的分析契约继续扩展。

备选方案：
- 一次性把“分析 + 迁移”一起做。
- 放弃原因是结构 schema 会更模糊，调试和展示都会更困难。

### 2. 使用标准化结构输出，而不是纯自然语言总结
分析结果应该收敛到固定字段，例如时长、镜头数估计、语音或字幕概览、段落列表、节奏标记等。

原因：
- 后续 OpenSpec change 需要可被程序直接消费的输出。
- 结构化结果更方便比较、可视化和验证。

备选方案：
- 仅保存一段 LLM 的自由总结文本。
- 放弃原因是很难复用，也很难向评审清晰解释系统到底提取了什么。

### 3. 第一阶段只要求两类结构维度
第一轮实现必须抽取：
- 脚本结构：hook、中段展开、结尾或 CTA
- 节奏结构：段落节奏、镜头密度、高潮位置等至少一种

原因：
- PRD 对 P0 的要求是三类结构中至少完成两类。
- 这样既满足评分要求，也能把第一阶段控制在可完成范围内。

备选方案：
- 同时把包装结构也做进去。
- 暂缓原因是第一阶段更需要闭环稳定，而不是维度铺太大。

### 4. 同时暴露人类可读和机器可读结果
分析结果应该同时以两种形态出现：
- 面向 UI 展示的结果摘要
- 面向后续流程复用的标准化 JSON 类结构

原因：
- 评审需要能看懂的结果页面。
- 后续迁移逻辑需要可确定、可依赖的字段。

备选方案：
- 只做页面展示结果。
- 放弃原因是后续功能会不得不从展示层再反向解析数据，成本更高。

### 5. 后端按 FastAPI + Service + Agent + Provider 分层预留
虽然当前 change 只覆盖样例分析，但后续完整系统预计会演进为 Python + FastAPI 驱动的 agent 后端，因此这一阶段需要提前预留清晰的接口边界。

建议分层如下：
- `api`：FastAPI 路由层，只处理请求、响应和鉴权边界
- `schemas`：Pydantic 输入输出结构，统一 API 契约
- `services`：业务编排层，负责“样例分析”“结构迁移”“缺口补全”等能力流程
- `agents`：面向复杂推理和多步决策的 agent 组件，例如分析 agent、迁移 agent、改片 agent
- `providers`：模型与外部能力适配层，例如多模态理解、ASR、TTS、AIGC 生成
- `repositories`：任务状态、分析结果、文件元数据等持久化接口

原因：
- 可以先完成单能力原型，再逐步替换为多 agent 编排，而不破坏 API。
- 可以避免把具体模型厂商、提示词和产品接口直接耦合在一起。
- 可以让前端始终依赖稳定的业务能力接口，而不是底层模型实现。

备选方案：
- 直接在 FastAPI 路由里编排模型调用。
- 放弃原因是后续一旦接入多 agent、多模型或异步任务，会非常难维护。

### 6. API 采用任务型异步接口而不是同步阻塞接口
考虑到视频分析、结构抽取和后续 agent 推理都可能是长耗时流程，后端 API 应优先采用 job-based 设计。

建议优先预留以下接口：
- `POST /sample-analysis/jobs`：创建样例分析任务
- `GET /sample-analysis/jobs/{job_id}`：查询样例分析任务状态与结果
- `POST /transfer/jobs`：创建结构迁移任务
- `GET /transfer/jobs/{job_id}`：查询结构迁移任务状态与结果
- `POST /asset-gap/jobs`：创建素材缺口识别任务
- `POST /asset-gap/fill`：执行素材缺口补全
- `POST /edit/iterate`：提交自然语言改片指令
- `GET /capabilities`：返回系统当前支持的能力、模型与策略
- `GET /health`：健康检查

原因：
- 异步任务模式更适合长耗时分析链路。
- 后续如果接入队列、SSE、WebSocket 或工作流引擎，不需要重做接口形态。
- 评审演示时也更容易展示“任务创建 -> 分析中 -> 结果完成”的完整流程。

备选方案：
- 所有接口同步返回最终结果。
- 放弃原因是用户体验差，也不利于扩展复杂 agent 流程。

### 7. 样例分析结果 schema 作为后续所有 agent 的共享输入
这一阶段定义的 `sample-analysis` 结果不应只是页面展示数据，而应成为后续结构迁移、缺口识别和自然语言改片的共享输入结构。

建议至少包含以下字段族：
- `job_id`、`status`、`sample_id`
- `metadata`
- `script_structure`
- `pace_structure`
- `confidence`
- `availability`
- `raw_outputs`

原因：
- 后续所有能力都需要建立在统一的样例理解结果之上。
- 固定 schema 后，可以先替换 provider，再替换 agent，而不用推翻前后端契约。

备选方案：
- 当前阶段只保存展示文本，后续再重新设计 schema。
- 放弃原因是会造成重复返工，并让后续能力之间难以衔接。

## Risks / Trade-offs

- [模型输出波动较大] -> 通过固定响应 schema 与字段校验来降低不稳定性。
- [镜头数与节奏指标可能只能估算] -> 在结果中明确标注为估计值，并记录推导来源。
- [多视频支持可能拉大第一轮复杂度] -> 先保证单视频跑通，再在同一契约下扩展到多视频。
- [原型代码与正式架构可能不一致] -> 把 `video_upload.py` 当作技术实验参考，而不是最终模块边界。
- [过早为完整 agent 系统预留过多抽象] -> 先固定 API 边界和 schema，复杂编排能力按后续 change 渐进落地。

## Migration Plan

当前不涉及生产迁移，这是一个从零开始的新能力。如果后续替换现有原型实现，应尽量保持分析输出 schema 稳定，再逐步把调用方迁移到新的服务边界上。

后端演进建议路径：
1. 先用单服务实现 `sample-analysis` 的最小闭环。
2. 再把模型调用逻辑下沉到 `providers` 层。
3. 当出现多步推理需求时，引入 `agents` 层做编排。
4. 当任务量和耗时上升后，再引入队列、缓存和任务持久化。

## Open Questions

- 字幕或语音概览应该优先来自 ASR、多模态模型，还是二者混合兜底？
- 上传后的样例元数据与分析结果最终落到什么持久化层？
- 是否要在 schema 中预留包装结构字段，即便这一轮暂时不填充？
- 异步任务状态更新是否需要在第一版就支持 SSE，还是先用轮询查询即可？
