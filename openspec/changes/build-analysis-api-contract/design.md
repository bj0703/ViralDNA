## Context

当前仓库已经实现了样例分析的最小后端闭环，但更多是为了证明“分析能力可跑通”。随着前端工作台、时间轴可视化和自然语言输入的加入，后端不能再只提供一个实验接口，而需要变成前后端都可依赖的稳定业务接口。

本次 change 的重点不是补新的分析算法，而是先固定契约。这样前端可以独立开发工作台，后端也可以在不破坏界面层的前提下，围绕当前已确认的豆包 HTTP 调用方式稳定演进。

当前已经明确的模型调用参数只有：
- `ARK_API_KEY`
- `ARK_ENDPOINT_ID`
- `ARK_BASE_URL`

并且当前请求方式已经确定为直接向 `chat/completions` 发送 HTTP POST 请求，而不是引入额外 SDK 模式、文件上传模式或更多配置项。因此本次 change 需要把这套边界清晰写死，避免实现阶段再次发散。

## Goals / Non-Goals

**Goals:**
- 定义面向前端的任务型分析 API。
- 定义自然语言意图识别的统一输入输出格式。
- 固定分析结果 schema 与前端展示模型。
- 明确前后端分离下的职责边界与状态流转。
- 固定当前豆包模型调用只依赖 `ARK_API_KEY`、`ARK_ENDPOINT_ID` 和 `ARK_BASE_URL`。

**Non-Goals:**
- 不在本次 change 中实现完整前端页面。
- 不在本次 change 中引入复杂工作流引擎、消息队列或真实多 agent 调度。
- 不在本次 change 中扩展结构迁移、素材缺口识别或结果视频生成的业务逻辑。
- 不在本次 change 中额外引入新的模型配置参数、SDK 上传链路或与 `chat/completions` 不一致的调用方式。

## Decisions

### 1. 后端 API 采用 job-based 异步模式
分析相关接口统一采用“创建任务 -> 查询状态/结果”的模式，而不是同步阻塞返回完整分析结果。

原因：
- 视频分析和后续 agent 推理天然是长耗时流程。
- 更适合前端工作台展示“待处理 / 分析中 / 完成 / 失败”等过程状态。
- 更容易扩展为队列、SSE 或 WebSocket。

### 2. 豆包调用方式固定为 chat/completions HTTP 请求
当前后端调用豆包模型时，只允许使用以下三个参数：
- `ARK_API_KEY`
- `ARK_ENDPOINT_ID`
- `ARK_BASE_URL`

其中：
- `ARK_API_KEY` 用于生成 `Authorization: Bearer ...`
- `ARK_ENDPOINT_ID` 作为请求体中的 `model`
- `ARK_BASE_URL` 直接指向 `https://ark.cn-beijing.volces.com/api/v3/chat/completions` 这一类完整请求地址

原因：
- 这已经是当前项目明确可用的请求方式。
- 可以避免实现时在 SDK 调用、文件上传模式和 HTTP 直调之间反复摇摆。
- 可以把配置层、provider 层和 API 契约保持一致。

备选方案：
- 保留额外配置项，预留多种调用模式。
- 放弃原因是当前阶段会增加实现歧义，并偏离你已确认的真实可用请求格式。

### 3. 区分结果接口与展示模型接口
后端同时提供：
- 原始标准化结果接口：供调试、导出和后续模块复用
- 前端展示模型接口：供时间轴、结构面板和播放器联动直接消费

原因：
- 前端页面不应自己拼装所有底层字段。
- 展示层可以保持稳定，即便底层 schema 小范围调整。

### 4. 自然语言意图识别作为独立接口
用户在右侧输入区提交的自然语言内容，先经过意图识别，再决定调用哪一类分析动作。

原因：
- 便于从“固定按钮触发”演进到“自然语言驱动工作台”。
- 可以把“视频分析、节奏分析、高光识别、导出结构”等动作统一收口。

### 5. 前端只依赖业务能力接口
前端不应感知 provider、prompt 或底层模型厂商，而只依赖稳定的分析 API 和 intent routing API。

原因：
- 可以独立替换底层实现。
- 可以保证前后端分离清晰，降低联调复杂度。

### 6. 配置层只暴露三个环境变量
后端配置层本次只允许暴露以下环境变量：
- `ARK_API_KEY`
- `ARK_ENDPOINT_ID`
- `ARK_BASE_URL`

原因：
- 可以保持配置和真实请求代码一一对应。
- 可以降低部署和联调时的理解成本。
- 可以避免后续实现中继续随意添加并未实际使用的参数。

## Risks / Trade-offs

- [过早抽象过多接口] -> 先围绕分析工作台的首批交互定义最小必需接口，不提前设计所有未来场景。
- [展示模型与原始结果可能重复] -> 通过清晰区分“复用结果”和“UI 聚合模型”控制复杂度。
- [意图识别不稳定] -> 第一版先限制为可控意图集合，并保留回退到默认全维度分析。
- [chat/completions 返回内容结构不稳定] -> 后端必须对模型原始输出做统一解析和兜底，而不能直接把原始文本透传给前端。
- [环境变量命名和现有代码不一致] -> 本次 apply 中统一只使用 `ARK_API_KEY`、`ARK_ENDPOINT_ID` 和 `ARK_BASE_URL`。

## Migration Plan

先保持当前样例分析接口可用，在新契约成熟后逐步迁移前端到新的标准路由和展示模型。若已有实验接口字段与新 schema 冲突，应优先兼容 `job_id`、`status` 和标准结果结构。

在实现层面，优先把现有测试请求收编为正式 provider，并保证：
- 配置只从 `ARK_API_KEY`、`ARK_ENDPOINT_ID` 和 `ARK_BASE_URL` 读取
- 请求统一走 HTTP `chat/completions`
- 后端对响应做标准化转换后再返回给前端
