## Context

当前动态意图规划完全依赖硬编码关键词匹配，无法理解用户复杂自然语言指令，扩展性差。第一版视频生成后，用户提出二次修改需求时必须全量重新跑所有6个Agent，浪费大量计算资源，等待时间很长。用户期望的流畅体验是"不满意就提一句新需求，系统自动快速出下一版视频"，不需要重头全跑。现有架构缺少状态版本管理和增量执行能力。

约束：
- 完全向后兼容，第一版全量流水线100%不受影响
- 不能引入外部大依赖
- LLM不可用时整个系统必须能自动降级继续工作

## Goals / Non-Goals

**Goals:**
1. LLM驱动智能意图规划，能理解任意用户自然语言指令，动态选择最优Agent执行子集
2. 共享记忆版本快照机制，每次新需求自动打版本号，保存完整历史状态
3. 增量编排引擎，自动检测已存在的缓存结果，直接跳过对应的Agent不重复执行
4. 新增Re-submit二次提交API，用户在已有Job基础上提交新需求快速生成下一版视频
5. 原有规则逻辑完整保留作为fallback兜底，零故障兜底机制

**Non-Goals:**
1. 本次不重构现有Agent执行拓扑，完全基于现有架构扩展
2. 不引入复杂的分布式任务调度系统
3. 不实现持久化数据库磁盘快照，全部在内存共享记忆中完成版本管理

## Decisions

| 决策 | 方案 | 理由 | 替代方案 |
|------|------|------|----------|
| LLM输出约束 | 要求LLM返回严格JSON格式，包含selected_agent_names/skip_reasons/confidence字段 | 结构化输出解析安全，结果可预测，完全避免LLM幻觉产生非法输出 | 自由文本输出后NLP解析，容错差不稳定 |
| 输出白名单校验 | LLM返回结果后必须经过AgentRegistry白名单过滤，不在注册表中的名称直接丢弃 | 防止LLM输出虚构的不存在Agent名称，绝对安全 | 不校验信任LLM输出，容易注入非法名称 |
| 置信度门限 | confidence < 0.6 时自动回退到原规则判断 | 低置信度场景直接用成熟规则兜底，避免LLM乱输出 | 无条件信任LLM输出，风险太高 |
| 增量跳过检测逻辑 | 执行Agent前先检查它所有write_keys是否都已在shared_memory中存在且非空 → 满足条件直接跳过 | 最简单最可靠的检测机制，零额外配置成本 | 人工配置每个Agent的输入变更条件，维护成本高 |
| 版本快照策略 | 每次新需求时深copy整个shared_memory字典，version字段自增+1 | 全量快照简单可靠，支持任意版本回溯 | 只做增量diff记录，实现复杂容易出错 |

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| LLM幻觉输出非法Agent名称 | 严格白名单校验+置信度过滤+规则三层兜底，三重保险 |
| 全量快照内存占用膨胀 | 每个job只保留最近3个版本历史，超过自动清理最旧的版本 |
| 增量跳过逻辑误跳过本该重跑的Agent | 在Re-submit接口中让用户可以勾选"强制重跑哪些Agent"，绕过自动检测 |
| 向后兼容性破坏 | 所有新功能都是可选扩展，完全不改动原有全量流水线执行路径 |
