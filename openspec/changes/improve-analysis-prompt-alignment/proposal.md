## Why

当前样例分析虽然已经接入了模型请求，但实际调用形态仍是把文件名、时长、备注和启发式结果拼成文本上下文后发送到 `chat/completions`。这只能得到“基于文本猜测的视频分析”，不等于真正对视频内容本身进行分析。

你已经提供了 `.env` 配置，本次工作的真实目标应当是：使用现有可用的模型 API，提交真实视频输入，让模型基于视频内容完成分析，而不是继续优化一条仅用于打通链路的伪多模态方案。

如果不先把这个目标写进 OpenSpec，后续实现很容易继续停留在“链路通了、返回了 JSON、前端能展示”的层面，而不是交付真正可用的视频理解能力。

## What Changes

- 将样例分析目标从“基于文本上下文推断视频内容”升级为“调用真实模型 API 分析视频内容”。
- 定义后端如何向目标模型 API 提交视频输入、分析指令和结构化输出约束。
- 明确当底层 API 不支持当前视频输入方式时，系统返回显式能力错误，而不是静默回退为文本猜测结果。
- 保留分析来源透明度能力，明确区分真实视频分析结果与任何降级结果。
- 更新前端与调试接口，使用户能确认本次任务是否真的走到了视频分析 API。

## Capabilities

### Modified Capabilities

- `reference-analysis-agent`: 从文本上下文推断模式升级为真实视频分析模式。
- `analysis-api`: 创建任务与返回结果时增加真实视频分析状态与来源追踪信息。
- `analysis-workbench-ui`: 工作台明确展示当前是否为真实视频分析结果。

## Impact

- 影响 `backend` 中模型请求适配层、样例分析 agent、provider 选择逻辑、错误处理和结果标准化逻辑。
- 影响 `frontend` 中状态展示与调试信息呈现。
- 会改变当前 fallback 策略：对于明确要求真实视频分析的请求，系统不能再把文本推断结果伪装成等价替代。
