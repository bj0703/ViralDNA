# 爆款结构迁移引擎 · 项目进度与评估

> 评估日期：2026-06-03
> 评估范围：阅读全仓代码、`openspec/`、`prd.md`、`prompt.md`、`RUN.md` 后整理
> 结论速览：**当前完成的是「样例理解 + 展示」这半个 P0 闭环；结构迁移、素材缺口、结果生成这另半个 P0 闭环尚未进入代码。最大风险是真实视频分析链路的 API 形态可能根本调不通。**

## 1. 项目当前形态

一句话：一个 **FastAPI 后端 + React/TS 前端工作台**，能上传样例视频、调用豆包模型（或本地启发式降级）做结构化拆解，并在三栏工作台里联动展示脚本结构、节奏结构和分析来源。

- 后端：`backend/app`，FastAPI，按 routes / services / providers / agents / repositories / schemas 分层，结构清晰。
- 前端：`frontend`，Vite + React + TS，三栏工作台（资源区 / 播放器+时间轴 / 分析面板），通过 Vite 代理打到后端。
- 模型：豆包 Ark，封装在 `ArkChatProvider`，未配置 key 时回退本地启发式 `HeuristicVideoAnalysisProvider`。
- 规划：`openspec/changes/` 下有 5 个变更，task 全部勾选完成。

## 4. 已完成部分的问题

按严重程度排序。

### 🔴 严重：真实视频分析链路的 API 形态很可能调不通

这是最关键的问题。`improve-analysis-prompt-alignment` 的目标是"真正把视频喂给模型"，但实现方式和官方参考原型对不上：

- 实现 [ark_chat.py](backend/app/providers/ark_chat.py#L36-L60)：走 `chat/completions`，把整段视频 **base64 内联**成 `{"type": "video_url", "video_url": {"url": "data:video/mp4;base64,..."}}`。
- 官方原型 [video_upload.py](video_upload.py)：走的是 **`responses` API**（`/api/v3/responses`），用 `client.files.create(...)` 先上传拿 `file_id`，再用 `{"type": "input_video", "file_id": ...}`，且能配 `fps` 抽帧。

两者是不同的 API、不同的消息结构。豆包的 `chat/completions` 能不能接 `video_url` 内联 base64 视频，强烈存疑——参考原型特地演示了"先上传文件"，恰恰说明视频不是这样内联传的。后果：

- 一旦真实视频请求被 endpoint 拒绝，`ReferenceAnalyzerAgentProvider` 的 `allow_fallback_on_model_error=False`（[dependencies.py](backend/app/dependencies.py#L18-L21) 用默认值）会让 `create_job` 直接抛 `RuntimeError → 502 VIDEO_ANALYSIS_FAILED`。前端会**直接报错，连降级结果都看不到**。
- 把整段视频 base64 塞进一次 `chat/completions`、`max_tokens` 只给 1200，对真实短视频很容易超长或被截断。
- **目前没有任何东西能证明"真实视频分析"真的成功过**。RUN.md 只说脚本"能跑通"，但跑通也可能是走了降级。

> 这条必须最先验证。建议立刻跑一次真实 endpoint，看 `provider_trace.used_real_video_input` 到底是不是 `true`。

### 🟠 中：结果丢失 + 无持久化

- 任务结果存在 `InMemorySampleAnalysisRepository`，**后端一重启全没**（RUN.md 第 11 节也承认）。演示时后端崩一次，之前所有 job 链接全失效。
- 上传的视频堆在 `backend/data/uploads/`（已有 9 个 UUID 目录的同一个测试视频），无清理、无大小限制。

### 🟠 中：映射层把真实分析"摊薄"回启发式骨架

即使模型成功返回，`_map_agent_output`（[reference_analyzer.py](backend/app/providers/reference_analyzer.py#L71-L141)）里：

- `pace_segments` 直接复用 heuristic 切出来的等分段（`fallback.pace_segments`），只把 pace 标签换成模型说法 —— 节奏分段的时间边界仍是启发式三等分，不是模型给的。
- `confidence` 是写死的常量（0.9 / 0.84 / 0.78），不反映模型实际置信度。
- `transcript_overview` 用的是 `upload.notes`，不是模型转写——所谓"转写概览"其实是用户自己填的备注。

也就是说，**前端展示的"真实分析结果"里掺了不少启发式占位值**，可解释性会在答辩时被质疑。

### 🟡 轻：包装结构未进入可视化

模型已经产出 `packaging_and_sound`（字幕密度、转场、BGM、封面风格）和 `migration_suggestion`，但前端 [App.tsx](frontend/src/App.tsx) 的面板只有 概览/脚本/节奏/风险/原始 五个 tab，**包装结构和迁移建议只能在"原始结果"里看 JSON**。PRD 任务2 的"包装结构"这一类其实模型分析了，却没展示出来，白白浪费。

### 🟡 轻：CORS 缺失 + 配置耦合

- 后端没配 CORS（RUN.md 第 6 节让用户"API Base URL 留空走 Vite 代理"绕过）。换个部署环境或前端直连就会失败。
- `ARK_ENDPOINT_ID` 在 [config.py](backend/app/core/config.py#L39-L47) 里硬编码了一个默认值 `ep-20260508213828-7ntjl`，和 spec 里"配置缺失就报错"的要求相悖——key 缺失会静默用默认 endpoint。

### 🟡 轻：意图识别是"只读"的，不驱动任何动作

`/sample-analysis/intent` 能识别 `pace-analysis` 等意图，但前端拿到结果只是**显示一个标签**（App.tsx 的 `intent-box`），不会真的去重新分析或筛选时间轴。意图路由目前是"识别了但没用上"。

### 🟢 提示：工程卫生

- `frontend/` 下有未追踪的构建产物 `vite.config.js` / `vite.config.d.ts` / `*.tsbuildinfo`，应进 `.gitignore`。
- `backend/data/uploads/` 的测试视频被纳入了仓库（应忽略）。
- 没有任何单元测试 / pytest；两个 `scripts/verify_*.py` 是手动比对脚本，不是自动化测试。

## 5. 后续应该做什么

按"先保命、再补闭环、最后冲分"的优先级排。

### 第一优先级 · 验证并坐实"真实视频分析"（1-2 天）

这是整个项目的地基，地基不稳后面全白做。

1. **先确认 endpoint 真实能力**。拿 `.env` 里的 endpoint 跑一次，确认 `chat/completions + 内联 base64 video_url` 到底支不支持。
2. 如果不支持（很可能），**按 [video_upload.py](video_upload.py) 的官方姿势改造 provider**：改用 `responses` API，`files.create` 上传拿 `file_id` → `wait_for_processing` → `input_video` 引用，并配 `fps` 抽帧。这意味着 `ArkChatProvider` 要拆出一个真正的 `video` 调用路径，而不是塞进 `chat_messages`。
3. **加一条端到端断言**：写个测试，跑真实样例后断言 `provider_trace.used_real_video_input == true`，让"是否真分析了视频"变成可自动验证的事，而不是靠人肉看 RUN.md。
4. 决定 fallback 策略：演示场景下建议给 `ReferenceAnalyzerAgentProvider` 开 `allow_fallback_on_model_error=True`，至少别让一次模型抖动就把整个 job 打成 502。

### 第二优先级 · 补齐 P0 的另一半闭环（这是拿分大头）

PRD 评分里"素材缺口处理"占 20 分，目前是 0。`prompt.md` 其实已经把 SlotMatcher / GapResolver / EditPlanner 的 prompt 设计好了，只是没写进代码。建议按 [build-reference-analyzer-agent](openspec/changes/build-reference-analyzer-agent/design.md) 里预留的多 agent 链路逐个落地：

1. **新内容/素材输入（任务3）**：加一个入口让用户提交"新主题 / 商品卖点 / 用户素材（图/视频/文案）"。这是迁移的"目标侧"，现在完全没有。
2. **AssetAnalyzerAgent（任务11）**：用 `prompt.md` 里的 `ASSET_INDEXER_SYSTEM_PROMPT` 分析用户素材，输出 content_type / tags / usable_segments。
3. **SlotMatcher + GapDetector（任务5）**：把样例结构的槽位和用户素材做匹配，标出缺口。`prompt.md` 的 `SLOT_MATCHER_PROMPT` 已经定义了 matches/gaps 输出。
4. **GapResolver（任务6）**：按 6 级优先级（reuse → static_graphic → text_card → brand_asset → ai_generate → ask_user）给出补全策略。先实现"文案/包装补全"这种纯文本的最省力，就能拿到任务6 的基础分。
5. **TransferPlanner / EditPlanner（任务4）**：产出 EditableTimeline 时间线草案。即使不真渲染成片，**一个能展示的分镜/时间线 JSON + 前端可视化**就能拿任务4、任务8 的分。

> 关键判断：成片渲染（FFmpeg/Remotion）很重，**性价比最高的是先做到"脚本 + 分镜 + 时间线草案 + 包装建议"这几样纯结构化产物**，PRD 任务4 明说"结果形式可为脚本/分镜/时间线/包装建议中任意两项"，不一定要成片。

### 第三优先级 · 让"迁移过程可视化"看得见（任务7，10 分）

这是答辩观感的核心。当前前端只能看"样例拆出了什么"。需要补一个**迁移视图**：左边样例结构槽位 → 中间映射箭头/缺口标记 → 右边新内容填充结果。把"哪里有缺口、用什么策略补的"显式画出来。这部分不需要新模型能力，主要是前端 + 把已有的 gap/match 数据渲染出来。

### 第四优先级 · 补已完成部分的短板

- **持久化**：把 `InMemorySampleAnalysisRepository` 换成 SQLite 或落盘 JSON，让 job 重启不丢（演示稳定性）。
- **暴露包装结构**：前端面板加"包装与声音""迁移建议"两个 tab，把模型已经分析出来、却被埋在 raw 里的 `packaging_and_sound` / `migration_suggestion` 展示出来——这是几乎零成本的加分。
- **修映射层的占位值**：让 `pace_segments` 用模型给的时间边界、`confidence` 用模型实际值，减少"真实结果里掺启发式"的可解释性风险。
- **意图识别接上动作**：让识别出的 `pace-analysis` 真的去过滤时间轴 / 触发对应面板，而不只是显示标签。这能顺带覆盖 PRD 任务12/13 的人机协同分。
- **配 CORS + 去掉硬编码 endpoint 默认值**：提升可部署性和配置正确性。

### 第五优先级 · 工程与交付物

- 把 `frontend` 构建产物、`backend/data/uploads/` 加进 `.gitignore`，清理仓库。
- 补 pytest 基础用例（至少覆盖 service 的 create_job、映射层、意图识别规则分支）。
- 走完 OpenSpec 流程：完成的变更 `archive` 到 `openspec/specs/`，让真值沉淀，别全堆在 `changes/`。
- 准备 PRD 要求的交付物：**项目说明文档（含整体 AI 架构、工具协议、安全边界）+ 演示视频 + 视频产物 Case**——目前这三样都没有。

## 6. 一句话总结

样例理解这条上游链路做得相当扎实（分层清晰、有 provider trace、有降级透明度），但项目离 PRD 的核心目标"**结构迁移**"还差一整条下游链路。当务之急有两件：**①坐实真实视频分析真的调通了**（否则地基是空的），**②把 prompt.md 里已设计好的 SlotMatcher/GapResolver/EditPlanner 从草稿变成代码**，补齐"新内容输入 → 缺口识别 → 迁移生成 → 可视化对比"这半个 P0 闭环。把这两件做完，分数能从目前的零散得分跳到一个完整可演示的闭环。
